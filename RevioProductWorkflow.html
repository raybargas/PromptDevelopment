The Rev.io Way

Pod-Based Product Development Across Three Realities
The Rev.io Way adapts to your development reality. Whether you're working with existing code, building from scratch, or navigating legacy systems, our pod-based approach ensures structured collaboration, clear documentation, and AI-powered development—with automatic progress tracking throughout.
The Rev.io Way: Always Evolving
This document represents our current approach and the tools we use today. However, change is the norm. The Rev.io Way is transformational by design—we continuously evaluate and adopt the most efficient tools and methodologies available. While Mary and Agent-OS serve us well now, we remain committed to staying on the cutting edge. As better solutions emerge, we will evolve our processes and transition our teams accordingly. The principles of structured collaboration, clear documentation, and AI-powered development remain constant; the specific tools and techniques will always adapt to serve us best.

Tools & Frameworks Powering The Rev.io Way

Our workflow leverages cutting-edge AI and automation tools. Here's what we use today:

AI Agents & Orchestration
  • BMAD / Mary: AI analyst that guides discovery, generates PRDs and PRFAQ documents through strategic elicitation
  • Agent-OS: Engineering orchestrator that generates technical specs, creates features/stories/tasks, and coordinates specialized sub-agents
Integrations & Connectivity (via MCP)

Agents interact with these tools autonomously or at our request—we don't access them directly

  • Model Context Protocol (MCP): Framework enabling seamless connections between AI agents and external systems
  • Notion MCP: Agents sync with Notion for PRD boards and progress tracking
  • GitHub MCP: Agents access repositories for code and documentation sync
  • Devin (DeepWiki): Agents research our Rev.io repositories to understand tech stacks, coding patterns, and standards
  • Context7: Agents retrieve version-specific, up-to-date documentation preventing API hallucinations and ensuring accurate code suggestions
  • Zen: Agents perform consensus gathering, contrarian analysis, and deep research
  • Playwright: Agents test their own work autonomously and iterate during builds
  • Figma: Agents reference UI/UX design mocks directly during implementation

Your Pod Structure

Every feature is developed by a pod of three roles working in close collaboration:

Product Manager Prompt Engineer Senior Engineer

Throughout this document, look for role badges to understand your specific responsibilities and when the pod collaborates as Team. Architecture Architect joins for technical foundation work and domain experts are involved in Minefield.

Most Prevalent
Brownfield Development
Working with existing code that needs modification, enhancement, or integration. Brownfield projects require understanding what exists before making changes. You'll survey the site with Devin to understand patterns, tech stack, and existing components. This is where most of your work happens.
1
Understand User Need & Define Solution
Product Manager
What Happens
  • PM presents the problem: What's the user's struggle? What's their pain point?
  • Team defines the solution: What would a better experience look like?
  • Pod has a conversation to align on requirements and approach
  • If scope is large enough or requirements unclear, PM may work with Mary for formal ideation
  • Product decides if PRFAQ is needed (usually only for large new features/modules)
Your Role as PM
Define the user problem and desired solution. Decide if formal discovery with Mary is needed. For large-scoped features (like the Project Management Module example), work with Mary and create PRFAQ. If PRFAQ is required, coordinate with stakeholders: schedule review meetings with executive leadership, ensure Director of Product and VP of Product are involved, facilitate the ratification process. For smaller modifications, skip formal discovery and move directly to surveying existing code.
2
Survey the Site with Devin
Team
What Happens
  • Identify all relevant repositories (API, SDK, frontend, components)
  • Feed Devin the problem and proposed solution
  • Devin performs deep read across repositories
  • Devin documents: user-facing features and workflows, component relationships, data models and schemas, existing patterns and conventions, tech stack details, dependencies across repos, potential ripple effects
  • Feed Devin findings to Mary/Agent-OS for context
Example: Adding Priority Field to Projects
Problem: Clients want to prioritize projects (High/Medium/Low)

Solution: Add priority dropdown to project form

Devin Survey (valuable to all roles):
  • For PM: Documents user-facing features, existing workflows, data relationships
  • For Engineering: Researches project schema in API repo, identifies existing form components in frontend repo, documents validation patterns, provides tech stack context (PostgreSQL, React, validation libraries)
  • For Team: Maps component relationships, identifies dependencies across repos, surfaces potential ripple effects

Result: Everyone understands what exists and how changes will impact the system
Your Role as Team
Identify relevant repos. Work with Devin to understand what exists. Document findings. Feed context to agents. Never modify existing code without understanding patterns first.
3
Create Specification
Prompt Engineer Senior Engineer
What Happens
  • Product Manager decides if modification is large enough to need sharded PRD
  • If large: Shard into numbered documents (01-, 02-, 03-...)
  • If small: Create spec directly
  • Feed requirements + Devin context to Agent-OS with /create-spec
  • Agent-OS generates features, stories, spec.md, tasks.md
  • Pod reviews spec together
Pod Collaboration: Spec Review
PM Prompt Engineer Senior Engineer

Review generated spec before implementation:
  • PM: Does this meet the user need? Is the solution correct?
  • Prompt Engineer: Are tasks clear? Is breakdown logical?
  • Senior Engineer: Does it align with existing patterns? Any risks?
Sharding Decision (Product Manager)
Product Manager decides if modification requires sharded PRD documents or just a spec. Large modifications (multiple features, significant refactoring) get sharded. Small changes (adding a field, modifying a component) go straight to spec. Radio up to Director of Product or VP of Product for guidance if scope is unclear.
4
Implementation & Tracking
Prompt Engineer Senior Engineer
What Happens
  • Engineer runs /implement-spec
  • Agent-OS orchestrates implementation using existing patterns
  • Progress updates automatically in Notion Progress Board
  • Senior Engineer monitors quality, runs verifiers, gates merges
Your Role as Senior Engineer
Ensure implementation follows existing patterns documented by Devin. Monitor quality. Run verifiers. Gate merges. Also perform Prompt Engineer duties: execute /create-spec and /implement-spec, monitor progress.
Ideal & Easy
Greenfield Development
Building new features from scratch with clean architecture. Greenfield is the ideal path—full discovery with Mary, PRFAQ ratification for large features, comprehensive documentation, and clear requirements. This is your happy path for new modules and major capabilities.
1
Discovery with Mary
Product Manager
What Happens
  • PM invokes Mary with /analyst
  • Mary conducts elicitation—asking strategic questions
  • PM answers about problem, users, goals, constraints, competitive landscape
  • PM incorporates insights from client focus groups, prospect feedback, competitive research
  • Mary creates ProjectBrief and PRFAQ
Example: Project Management Module
Client Context: Multiple prospects said "We need project management in your PSA" and clients are using external tools creating workflow friction.

PM brings to Mary: "We need integrated project management in Rev.io PSA. Clients are losing visibility as projects scale."

Mary elicits: Who are the end users? What are they trying to accomplish? What does success look like?

Mary generates: ProjectBrief + PRFAQ documenting vision, customer value prop, competitive positioning
Critical Gate: PRFAQ Ratification
For large-scoped features (brand new modules), PRFAQ must be ratified by:
  • Executive leadership (CEO, COO)
  • Product leadership
  • Sales/success teams

Process: PM presents PRFAQ, gathers feedback, iterates with Mary until ratified. This validates customer value before building anything.
Your Role as PM
Lead discovery with external and internal stakeholders. Work with Mary to document vision. If PRFAQ is needed, coordinate review meetings: schedule with stakeholders (executive leadership, product leadership, sales/success teams), ensure Director of Product and VP of Product are involved, facilitate ratification process with Mary iterations. For smaller features, skip PRFAQ and proceed directly to spec creation.
2
Project Formation & PRD Creation
Team
What Happens

Two parallel work streams:

PM Stream: Create PRD
PM works with Mary
  • Mary creates full PRD (problem, requirements, epics, success criteria)
  • Team shards PRD into numbered documents if scope is large
  • Validate sharded documents
Engineering Stream: Set Up Infrastructure
Architect Senior Engineer Prompt Engineer

Set up Notion boards and Agent-OS config:
  • PRD Board (Notion → GitHub sync)
  • Progress Board (GitHub → Notion sync)
  • product/mission.md, product/roadmap.md, product/tech-stack.md
  • Standards from templates
Your Role as PM
Work with Mary to create PRD. Decide if PRD needs sharding (radio up to Director/VP of Product for guidance if unclear). Collaborate with team to shard if needed. Hand off to engineering when ready.

Most Challenging
Minefield Development
Working with legacy, brittle code that carries high risk. Minefield areas are 5+ years old, are brittle, or support critical business functions (billing, payments). Architecture and domain experts must be heavily involved. Re-architecting is an explicit option—sometimes the last time you modify before rewriting completely.
1
Understand Constraint & Define Solution
Product Manager Architect
What Happens
  • PM presents the problem: What's the user's struggle?
  • Architecture identifies this as Minefield: What can and can't be done?
  • Domain experts weigh in on constraints and ripple effects
  • Team aligns on solution approach
  • Decision point: Modify existing or re-architect?
Re-Architect Decision Point
Critical question: Is this the last time?

If you'll need to modify this again in 3-6-9 months, consider re-architecting now. This spins off into either:
  • PRFAQ for large re-architecture
  • Smaller-scale PRD for targeted rewrite

Architecture leads this decision with domain expert input.
Your Role as PM
Define user problem and desired solution. Work with Architecture to understand constraints. Support the re-architect decision if needed.
2
Survey Site with Devin + Architecture
Team Architect
What Happens
  • Identify all relevant repositories
  • Devin performs deep read across codebase documenting: legacy patterns, brittle dependencies, critical business logic, constraint boundaries, integration points, data flows
  • Architecture reviews Devin findings
  • Domain experts validate constraints and dependencies
  • Document ripple effects of potential changes
  • Feed comprehensive context to agents
Critical: Humans Heavily in the Loop
Architecture and domain experts must validate Devin findings. This is NOT autonomous. Review patterns, constraints, and dependencies before proceeding.
Your Role as Architect
Lead site survey. Validate Devin findings. Identify constraints. Document ripple effects. Guide re-architect decision if needed.
3
Create Specification
Prompt Engineer Senior Engineer Architect
What Happens
  • Team creates spec with heavy architecture review
  • Feed requirements + Devin context + architecture guidance to Agent-OS
  • Agent-OS generates spec respecting constraints
  • Architecture and domain experts review
  • Extra validation gate before proceeding
Extra Gate: Minefield Review
Before implementation:
  • Senior Engineer reviews architecture alignment
  • Domain experts validate business logic changes
  • Architecture confirms ripple effects are acceptable
  • Team agrees to proceed
Your Role as Senior Engineer
Lead spec review with Architecture. Ensure constraints are respected. Validate ripple effects. Gate before implementation.
4
Implementation with Human Oversight
Prompt Engineer Senior Engineer Architect
What Happens
  • Engineer runs /implement-spec
  • Senior Engineer monitors closely
  • Architecture reviews critical changes
  • Domain experts validate business logic
  • Progress updates in Notion
  • Extended quality gates before merge
Extended Quality Gate
Before merge approval:
  • Senior Engineer runs all verifiers
  • Architecture reviews critical paths
  • Domain experts validate business outcomes
  • Team agrees changes are safe
Your Role as Architect
Review critical implementation changes. Validate architecture integrity. Ensure Minefield remains stable. Approve final merge.

The Bottom Line
Each pod member focuses on their expertise. PM drives vision and validates customer value. Prompt Engineer orchestrates AI development. Senior Engineer ensures quality and performs implementation duties. Architecture defines technical foundation and guides Minefield work. Mary handles documentation. Agent-OS handles implementation. Devin surveys existing code. The pod focuses on building the right thing, the right way, for our clients—whether in Brownfield, Greenfield, or Minefield.
