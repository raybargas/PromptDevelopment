The Rev.io Way

Pod-Based Product Development Across Three Realities
The Rev.io Way adapts to your development reality. Whether you're working with existing code, building from scratch, or navigating legacy systems, our pod-based approach ensures structured collaboration, clear documentation, and AI-powered development—with automatic progress tracking throughout.
The Rev.io Way: Always Evolving
This document represents our current approach and the tools we use today. However, change is the norm. The Rev.io Way is transformational by design—we continuously evaluate and adopt the most efficient tools and methodologies available. While Mary and Agent-OS serve us well now, we remain committed to staying on the cutting edge. As better solutions emerge, we will evolve our processes and transition our teams accordingly. The principles of structured collaboration, clear documentation, and AI-powered development remain constant; the specific tools and techniques will always adapt to serve us best.

Tools & Frameworks

Core Tools: Product Manager Mary (discovery, PRDs, PRFAQs) · Engineers Agent-OS (Shape Spec → Write Spec → Create Tasks → Implement → Orchestrate), Context7, Playwright · Team Devin (repo research), Notion (sync), Zen (research), Figma

All tools require local MCP configuration. Devin outputs must be reviewed by the pod member using it.

Your Pod Structure

Every feature is developed by a pod of three roles working in close collaboration:

Product Manager Prompt Engineer Senior Engineer

Throughout this document, look for role badges to understand your specific responsibilities and when the pod collaborates as Team. Architecture Architect joins for technical foundation work and domain experts are involved in Minefield.

Notion is Your Source of Truth
PRFAQs, PRDs, and sharded PRDs live in Notion. PMs edit in Notion → syncs to GitHub → Engineers read from GitHub. Progress syncs back to Notion automatically. No manual handoffs, no status meetings.
The Agent-OS Workflow (6 Phases)
Phase 0 (Pre-work): Plan Product with Mary (discovery, PRFAQ, PRD)
Phase 1: Shape Spec — Gather visual assets, requirements, Devin context
Phase 2: Write Spec — Agents analyze UI patterns, map to code patterns
Phase 3: Create Tasks — Break spec into executable tasks with visual references
Phase 4: Implement Tasks — Generate code from visuals + standards, verify with Playwright
Phase 5: Orchestrate Tasks — Coordinate delivery, QA, visual verification

Runtime Modes: How Agent-OS Adapts to Context

Brownfield, Greenfield, and Minefield aren't just situational realities—they're runtime configurations that adjust how the agentic system behaves. Agent-OS automatically adapts its autonomy level, verification depth, and context handling based on which mode is active.

Mode Agent Behavior Verification Depth Autonomy Required Roles Approval Gates
🟤 Brownfield Contextual + Diff-Based
Devin gathers patterns, AST diff, schema introspection		 Medium
Diff + regression focused		 Medium
Proposes with justification		 PM, Prompt Eng, Sr Eng Standard review
🟢 Greenfield Generative + Blueprint-Based
Mary discovery, schema-first, scaffolds from templates		 Low
Standards compliance		 High
Human reviews architecture		 PM, Prompt Eng Lightweight gates
🔴 Minefield Supervised + Rollback-Enabled
Restricted mode, shadow branches, sandbox testing		 High
Full validation + expert review		 Low
Assistive, not generative		 PM, Sr Eng, Architect Multi-stage validation

The orchestrator switches configurations automatically based on the mode. This ensures the right balance of speed, safety, and human oversight for each context.

1
Understand User Need & Define Solution
Product Manager
What Happens
  • PM presents the problem: What's the user's struggle? What's their pain point?
  • Team defines the solution: What would a better experience look like?
  • Pod has a conversation to align on requirements and approach
  • If scope is large enough or requirements unclear, PM may work with Mary for formal ideation
  • Product decides if PRFAQ is needed (usually only for large new features/modules)
Your Role as Product Manager
Define problem and solution. Large features → work with Mary, create PRFAQ, coordinate stakeholder ratification. Small mods → skip formal discovery.
2
Survey the Site with Devin
Team
What Happens
  • Identify all relevant repositories (API, SDK, frontend, components)
  • Access Devin via MCP (requires local configuration by pod member)
  • Feed Devin the problem and proposed solution
  • Devin performs deep read across repositories
  • Devin documents: user-facing features and workflows, component relationships, data models and schemas, existing patterns and conventions, tech stack details, dependencies across repos, potential ripple effects
  • All Devin outputs must be reviewed by the pod member leveraging the tool
  • Technical findings reviewed by Prompt Engineer and Senior Engineer with Architecture guidance as needed
  • Product-level findings reviewed by Product Manager
  • Feed reviewed Devin findings to Mary/Agent-OS for context
Example: Adding Priority Field to Projects
Problem: Clients want to prioritize projects (High/Medium/Low)

Solution: Add priority dropdown to project form

Devin Survey (valuable to all roles):
  • For PM: Documents user-facing features, existing workflows, data relationships
  • For Engineering: Researches project schema in API repo, identifies existing form components in frontend repo, documents validation patterns, provides tech stack context (PostgreSQL, React, validation libraries)
  • For Team: Maps component relationships, identifies dependencies across repos, surfaces potential ripple effects

Result: Everyone understands what exists and how changes will impact the system
Your Role as Team
Identify repos → Devin survey → Review findings → Feed to agents. Never modify code without understanding patterns first.
3
Create Specification
Prompt Engineer Senior Engineer
What Happens
  • Product Manager decides: sharded PRD (large) or direct spec (small)
  • Shape Spec: Gather visual assets if UI changes are involved
  • Write Spec: Feed requirements + Devin context + visual assets to Agent-OS
  • Create Tasks: Agent-OS breaks spec into tasks with visual references
  • Team reviews spec → Product Manager validates product/UX → iterate until approved
Spec Review Checklist
Product Manager: User need met? · Prompt Engineer: Tasks clear? · Senior Engineer: Aligns with patterns? · Radio up to Director/VP Product if scope unclear.
4
Implementation & Tracking
Prompt Engineer Senior Engineer
What Happens
  • Implement Tasks: Engineer runs implementation—agents use visual references + standards to generate code
  • Agents may use Playwright MCP to check UI in live browser
  • Iterative refinement with /improve-ui commands or visual verification
  • Orchestrate Tasks: Coordinate delivery, QA, and verification
  • Visual verification compares rendered output to mockups (like snapshot testing, AI-enhanced)
  • Progress updates automatically in Notion Progress Board
  • Senior Engineer monitors quality, runs verifiers (AI agents that check quality of completed work), gates merges
Your Role as Senior Engineer
Follow Devin-documented patterns. Run verifiers (AI quality checks). Gate merges. Perform Prompt Engineer duties as needed.

Most Challenging
Minefield Development
Working with legacy, brittle code that carries high risk. Minefield areas are 5+ years old, are brittle, or support critical business functions (billing, payments). Architecture and domain experts must be heavily involved. Re-architecting is an explicit option—sometimes the last time you modify before rewriting completely.
🔴 Minefield Runtime Configuration
Agent Behavior: Supervised + Rollback-Enabled
Autonomy Level
Low — System is assistive, not generative. No autonomous commits
Verification Focus
Full Validation — Dual validation (Senior Engineer + Architect), expert review required
Context Handling
Restricted Mode — Shadow branches, rollback checkpoints, sandbox testing
Approval Gates
Multi-stage validation — Extended gates with Architect and domain expert sign-off

The Bottom Line

Pod Collaboration Model: RACI Chart

This chart defines responsibility and accountability for key activities across the workflow. R = Responsible (does the work), A = Accountable (single owner of outcome), C = Consulted (provides input), I = Informed (kept in the loop).

Activity Product Manager Prompt Engineer Senior Engineer
Problem Definition & User Need R/A Defines problem, articulates user pain, decides solution approach C Provides input on technical feasibility C Provides input on technical feasibility and patterns
Discovery & PRFAQ (Greenfield) R/A Works with Mary, coordinates stakeholder ratification C Consulted on scope and vision C Consulted on scope and vision
PRD Creation & Sharding R/A Creates PRD with Mary, decides sharding strategy R Assists PM in creating PRD and ensures validity C Reviews technical approach and architecture
Architecture & Pattern Decision I Informed of technical approach C Provides input on patterns and standards R/A Decides architecture, patterns, and technical standards
Shape Spec (Visual Assets) C Provides mockups or confirms UX direction R/A Gathers visual assets from Figma/mockups C Reviews asset completeness
Write Spec (Agent-OS Input) I Informed of spec progress R/A Feeds PRD + context + visuals to Agent-OS R Co-authors spec, ensures pattern alignment
Create Tasks (Task Breakdown) I Reviews task structure R/A Works with Agent-OS to break down spec R Validates task structure and dependencies
Spec Review & Approval A Final approval on product/UX alignment C Confirms task clarity and executability R Validates technical approach and patterns
Implementation (Code Generation) I Informed of progress R Runs Agent-OS implementation, iterates R/A Monitors quality, validates patterns

Quality Verification (Verifiers) I Informed of quality status C Assists with verification issues R/A Runs AI verifiers, gates merges

Overlapping Collaboration: Throughout the workflow, Prompt Engineer and Senior Engineer work closely together on spec creation and implementation, with Senior Engineer performing Prompt Engineer duties as needed. Product Manager stays informed through progress updates, stepping in for approvals and validation checkpoints.
